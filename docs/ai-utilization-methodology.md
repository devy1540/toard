# AI 활용 지수 방법론

## 1. 버전과 상태

- 방법론 버전: `utilization-v1`
- 제품 표시: `AI 활용 지수 Beta`
- 상태: 초기 방법론
- 상위 정책: [AI 활용 지수 정책](ai-utilization-policy.md)

이 방법론은 사용자의 현재 AI 활용 메타데이터를 본인의 과거 분포와 비교한다. 점수는 생산성이나 결과 품질이 아니라 관측 가능한 활용 흐름의 상대적 변화다.

## 2. 접근 방식

### 채택: 자기 기준 다축 지수

- 개인의 과거 28일을 기준선으로 사용한다.
- 현재 7일의 세부 축을 기준선의 일별 분포와 비교한다.
- 기준선 중앙값을 50점으로 둔다.
- 계산 가능한 세부 축이 2개 이상일 때만 종합점수를 제공한다.
- 원시 비율과 신뢰도를 점수와 함께 제공한다.

### 채택하지 않음: 절대 기준 지수

모든 사람에게 같은 성공률·사용량 기준을 적용하면 업무와 provider 차이를 무시하고 개인 간 비교 점수로 오해되기 쉽다.

### 채택하지 않음: 사용량 성숙도 점수

활성 일수, 토큰, 세션, 도구 호출, 도구 종류를 많이 사용할수록 점수가 오르는 방식은 불필요한 사용을 유도하므로 채택하지 않는다.

## 3. 기간

- 현재 기간: 계산 시점 직전의 완료된 7개 캘린더 일
- 기준 기간: 현재 기간 직전의 완료된 28개 캘린더 일
- 일 경계: 조직 타임존
- 계산 시점: 현재 시각 이후의 부분 일은 제외

예를 들어 조직 타임존이 `Asia/Seoul`이고 7월 15일 14시에 계산하면 현재 기간은 7월 8일 00:00 이상 7월 15일 00:00 미만, 기준 기간은 6월 10일 00:00 이상 7월 8일 00:00 미만이다.

첫 버전은 사용자 선택 기간에 따라 산식을 바꾸지 않는다. 화면의 일반 인사이트 기간 필터와 별개로 고정된 7일/28일 기간을 명시한다.

## 4. 입력 데이터

### 4.1 사용량 이벤트

- `sessionId`
- `providerKey`
- `model`
- `ts`
- `inputTokens`
- `cacheReadTokens`
- `cacheCreationTokens`

`outputTokens`와 `costUsd`는 설명용 관측값으로 보여줄 수 있지만 점수에는 포함하지 않는다.

### 4.2 도구 활동 이벤트

- `sessionId`
- `ts`
- `activityKind`
- `itemKey`
- `outcome`
- `detection`

`displayName`, `pluginKey`, host는 점수 계산에 사용하지 않는다.

### 4.3 콘텐츠 독립성

프롬프트 레코드, 응답 본문, 코드, 도구 인자·출력은 읽지 않는다. 콘텐츠 수집 활성 여부는 계산 입력과 신뢰도 입력에서 모두 제외한다.

## 5. provider 지원 범위

각 adapter는 다음 capability를 명시해야 한다.

- `reportsCacheRead`: 캐시 읽기 토큰을 의미 있게 제공하는가?
- `reportsToolOutcome`: 도구 성공·실패를 구분해 제공하는가?
- `reportsSessionId`: 세션 식별자를 안정적으로 제공하는가?

capability가 없는 신호는 0으로 간주하지 않는다. 해당 provider 이벤트는 관련 축의 분자와 분모에서 제외하고 데이터 범위에 `unsupported`로 기록한다.

## 6. 일별 원시 지표

### 6.1 컨텍스트 연속성

캐시로 재사용된 입력 컨텍스트의 비율이다.

```text
contextDenominator = inputTokens + cacheReadTokens + cacheCreationTokens
contextContinuity = cacheReadTokens / contextDenominator
```

- `reportsCacheRead=true`인 이벤트만 포함한다.
- 분모가 0인 일은 계산하지 않는다.
- 값 범위는 0~1이다.
- 높은 값은 컨텍스트 재사용 비중이 높다는 의미일 뿐 업무 품질이 높다는 의미가 아니다.

### 6.2 실행 안정성

결과가 확인된 MCP·스킬 호출 중 성공 비율이다.

```text
knownCalls = successes + failures
executionStability = successes / knownCalls
knownOutcomeCoverage = knownCalls / (successes + failures + unknown)
```

- `outcome=unknown`은 성공률 분모에서 제외한다.
- unknown이 많은 경우 점수가 좋아지는 것을 막기 위해 결과 확인 범위를 별도로 요구한다.
- 값 범위는 0~1이며 높을수록 확인된 실행 실패가 적다.

### 6.3 복구 부담

같은 세션에서 같은 도구가 실패한 뒤 짧은 시간 안에 다시 실패한 비율이다.

도구 이벤트를 사용자, 세션, `activityKind`, `itemKey`, 시각, dedup key 순서로 정렬한다. 다음 조건을 모두 만족하는 실패를 반복 실패로 센다.

1. 현재 이벤트가 `failure`다.
2. 같은 세션·종류·도구의 직전 결과 확인 이벤트도 `failure`다.
3. 두 이벤트의 간격이 30분 이하다.

```text
recoveryBurden = repeatedFailures / knownCalls
```

- `sessionId`가 없는 이벤트는 반복 실패 판정에서 제외한다.
- 값 범위는 0~1이며 낮을수록 반복 실패 부담이 적다.

## 7. 중립 관측값

다음 값은 화면에 표시할 수 있지만 종합점수와 세부 축 점수에는 포함하지 않는다.

- 도구를 사용한 세션 비율
- 사용한 MCP·스킬 종류 수
- 활성 일수
- provider·model 구성
- 세션, 이벤트, 토큰, 비용 총량
- host·기기 수

이 값에는 높음·낮음의 우열 방향을 부여하지 않는다.

## 8. 계산 가능 조건

### 8.1 개인 공통 조건

- 현재 기간 활성일 3일 이상
- 현재 기간 유효 세션 5개 이상
- 기준 기간 활성일 7일 이상

공통 조건을 충족하지 않으면 종합점수를 계산하지 않는다.

### 8.2 세부 축 조건

컨텍스트 연속성:

- 현재 기간 계산 가능 일 3일 이상
- 기준 기간 계산 가능 일 7일 이상
- 현재·기준 기간의 지원되는 컨텍스트 분모가 각각 0보다 큼

실행 안정성:

- 현재 기간 결과 확인 호출 10회 이상
- 기준 기간 결과 확인 호출 10회 이상
- 현재·기준 기간 `knownOutcomeCoverage >= 0.70`
- 기준 기간 계산 가능 일 7일 이상

복구 부담:

- 실행 안정성과 같은 호출·coverage 조건
- 현재·기준 기간에 세션 ID가 있는 결과 확인 호출 10회 이상

조건을 충족하지 못한 축은 0점이 아니라 `unavailable`이다.

## 9. 자기 기준 정규화

각 축에서 현재 값은 현재 기간의 계산 가능한 일별 비율 중앙값이다. 기준선은 기준 기간의 계산 가능한 일별 비율 배열이다.

```text
baselineMedian = median(baselineDailyValues)
mad = median(abs(value - baselineMedian))
robustScale = max(1.4826 * mad, 0.05)
```

`0.05`는 비율 지표의 기준선 변동이 0에 가까울 때 작은 차이가 과도한 점수 차이로 증폭되는 것을 막는 5%p 최소 스케일이다.

방향은 다음과 같다.

- 컨텍스트 연속성: `+1`
- 실행 안정성: `+1`
- 복구 부담: `-1`

```text
robustZ = direction * (currentValue - baselineMedian) / robustScale
dimensionScore = clamp(round(50 + 15 * robustZ), 0, 100)
```

- 50점은 본인의 기준선 중앙값이다.
- 약 1 robust scale 차이는 15점 차이다.
- 0점과 100점은 상대 척도의 하한·상한이지 실패·완벽을 뜻하지 않는다.

## 10. 종합점수

- 유효한 세부 축이 2개 이상일 때만 계산한다.
- 유효한 축을 동일 가중치로 산술평균하고 반올림한다.
- 계산되지 않은 축은 0점으로 넣지 않고 가중치에서 제외한다.

```text
overallScore = round(sum(validDimensionScores) / validDimensionCount)
```

특정 축에 임의의 높은 가중치를 부여하지 않는다. 실제 사용자 데이터와 해석 오류를 검증한 뒤 가중치를 변경하려면 방법론 버전을 올린다.

## 11. 신뢰도

신뢰도는 점수와 별도로 계산한다.

### 높음

- 세 축 모두 계산 가능
- 기준 기간 활성일 14일 이상
- 도구 결과 확인 범위 90% 이상
- 지원되지 않는 사용량 이벤트가 전체 이벤트의 10% 이하

### 보통

- 두 축 이상 계산 가능
- 기준 기간 활성일 7일 이상
- 계산에 사용된 도구 축의 결과 확인 범위 70% 이상

### 낮음

낮은 신뢰도에서는 종합점수를 제공하지 않는다. 원시 관측값과 계산 불가 이유만 표시한다.

## 12. 해석

점수는 등급으로 표시하지 않는다. 화면 문구는 다음 원칙을 따른다.

- 45~55: `본인의 평소 범위`
- 56 이상: `본인의 평소보다 높음`
- 44 이하: `본인의 평소보다 낮음`

이 구간은 색상·문구를 위한 표시 규칙이며 별도 방법론 점수 구간이나 성과 등급이 아니다. 빨강·초록만으로 좋고 나쁨을 표현하지 않는다.

기간 간 변화가 5점 미만이면 `유지`, 5점 이상이면 방향만 설명한다. 서로 다른 방법론 버전은 비교하지 않는다.

## 13. 조직 집계

조직 집계에는 같은 방법론 버전이고 신뢰도가 보통 이상인 개인 결과만 포함한다.

- 활성 사용자 5명 미만: `suppressed`
- 5명 이상: 개인 행을 버린 뒤 다음 값만 반환
  - 종합점수 중앙값
  - 종합점수 25·75 백분위
  - 세부 축별 중앙값
  - 직전 동등 기간 대비 `상승`, `유지`, `하락` 비율
  - 포함 인원 수와 제외 인원 수

포함·제외 인원 수 자체가 5명 미만 하위 집단을 드러내는 필터에서는 세부 제외 사유를 반환하지 않는다.

## 14. 이유 코드

- `insufficient_current_days`
- `insufficient_current_sessions`
- `insufficient_baseline_days`
- `unsupported_cache_signal`
- `insufficient_context_days`
- `insufficient_known_tool_calls`
- `low_tool_outcome_coverage`
- `insufficient_session_tool_calls`
- `insufficient_valid_dimensions`
- `suppressed_small_cohort`
- `mixed_methodology_versions`

사용자 화면은 이유 코드를 번역된 설명으로 표시하고 내부 구현 세부사항이나 식별자를 노출하지 않는다.

## 15. 계산 예시

기준 기간 실행 안정성 중앙값이 0.80, MAD가 0.04이고 현재 값이 0.90이라고 가정한다.

```text
robustScale = max(1.4826 * 0.04, 0.05) = 0.059304
robustZ = (0.90 - 0.80) / 0.059304 = 1.686
score = round(50 + 15 * 1.686) = 75
```

75점은 본인의 평소 실행 안정성보다 약 1.69 robust scale 높다는 뜻이다. 다른 사용자보다 우수하다는 뜻이 아니다.

## 16. 알려진 한계

- 코드 수락, 테스트 성공, PR 완료, 업무 결과를 관측하지 않는다.
- 높은 캐시 재사용률이 모든 업무에서 바람직하다고 단정할 수 없다.
- 도구를 사용하지 않는 작업은 도구 축을 계산할 수 없다.
- provider별 로그 형식과 outcome 지원 차이로 신뢰도가 달라질 수 있다.
- 자기 기준 방식은 신규 사용자와 사용 패턴이 급격히 바뀐 사용자에게 기준선 준비 기간이 필요하다.
- 표본 5명 억제는 차등 개인정보 보호가 아니다.
