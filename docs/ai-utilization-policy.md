# AI 활용 지수 정책

## 문서 상태

- 상태: 승인됨
- 적용 대상: `AI 활용 지수 Beta`와 그 파생 개인·조직 화면
- 정책 소유자: toard 프로젝트
- 방법론 문서: [AI 활용 지수 방법론](ai-utilization-methodology.md)
- 구현 설계: [AI 활용 지수 설계](superpowers/specs/2026-07-15-ai-utilization-index-design.md)

이 문서는 AI 활용 지수를 계산하고 표시하는 모든 구현보다 우선하는 규범 문서다. 방법론과 구현은 이 정책의 공개 범위, 금지 용도, 데이터 경계를 완화할 수 없다.

## 1. 목적

AI 활용 지수는 사용자가 자신의 AI 활용 습관을 이해하고 개선 기회를 찾도록 돕는 개인 코칭 지표다. 관측 가능한 사용 메타데이터를 요약할 뿐 생산성, 업무 성과, 코드 품질, 개인의 역량을 측정한다고 주장하지 않는다.

지수는 다음 질문에 답하는 데 사용한다.

- 최근 AI 활용 흐름이 본인의 평소 패턴과 비교해 어떻게 달라졌는가?
- 컨텍스트 재사용, 도구 실행 안정성, 반복 실패 부담 중 어디에 변화가 있었는가?
- 현재 점수를 신뢰할 만큼 데이터가 충분한가?

다음 질문에는 답하지 않는다.

- 누가 AI를 가장 잘 사용하는가?
- 누가 가장 생산적인가?
- 특정 업무의 결과가 좋은가?
- 더 많은 토큰이나 도구 호출이 더 좋은가?

## 2. 용어

- **개인 지수**: 한 사용자의 메타데이터를 그 사용자의 과거 기준선과 비교한 값.
- **세부 축**: 컨텍스트 연속성, 실행 안정성, 복구 부담처럼 서로 다른 관측 신호.
- **기준선**: 현재 기간보다 앞선 본인 데이터로 계산한 비교 분포.
- **신뢰도**: 점수 계산에 사용된 데이터 양, 지원 범위, 결측 정도를 나타내는 별도 상태.
- **활성 사용자**: 집계 대상 기간에 유효한 사용량 세션이 하나 이상 있는 사용자.
- **조직 집계**: 개인별 행을 반환하지 않고 동일 방법론 버전의 개인 지수를 통계량으로만 요약한 값.
- **표본 억제**: 집계 대상 활성 사용자가 5명보다 적을 때 값을 계산하거나 반환하지 않는 처리.

## 3. 허용 용도

- 개인이 자신의 기간별 활용 패턴과 세부 축을 확인하는 기능
- 개인에게 데이터에 근거한 개선 힌트를 제공하는 기능
- 조직이 익명 집계를 통해 공통 교육·도구 지원 필요성을 파악하는 기능
- 방법론 품질, 데이터 완전성, provider 지원 범위를 검증하는 내부 운영
- 사용자가 요청한 데이터 오류 조사와 재계산

## 4. 금지 용도

활용 지수와 그 세부 축은 다음 용도로 사용할 수 없다.

- 인사평가, 보상, 승진, 채용, 징계, 해고
- 개인·팀 순위, 경쟁표, 목표 점수, 최소 점수 할당
- 근무시간, 업무 태도, 성실성, 개인 역량 추정
- 개인별 AI 사용량 강제 또는 모델·도구 사용 제한
- 기존 비용·토큰 순위와 활용 지수를 결합한 개인 평가
- 충분하지 않은 데이터를 0점으로 바꾸어 불리하게 해석하는 행위
- 다른 방법론 버전의 점수를 같은 척도처럼 비교하는 행위

관리 UI, API, 내보내기 기능도 위 용도를 가능하게 하는 개인별 활용 지수 목록을 제공해서는 안 된다.

## 5. 데이터 경계

### 5.1 허용 데이터

첫 버전은 다음 메타데이터만 사용할 수 있다.

- 세션 식별자와 이벤트 발생 시각
- provider와 model
- 입력, 출력, 캐시 읽기, 캐시 생성 토큰
- MCP·스킬 식별자와 `success`, `failure`, `unknown` 결과
- 위 메타데이터에서 계산한 일별 횟수, 비율, 기준선, 신뢰도

host 정보는 데이터 수집 누락과 지원 범위를 설명하는 데만 사용할 수 있으며 점수를 높이거나 낮추는 요인으로 사용하지 않는다.

### 5.2 금지 데이터

다음 데이터는 수집 여부와 관계없이 활용 지수 계산에 사용할 수 없다.

- 프롬프트·응답·코드 본문
- 파일 내용, 파일 경로, 저장소 이름, 작업 디렉터리
- 명령어, 도구 인자, 도구 실행 결과
- E2EE 히스토리 복호화 결과
- 질문 주제, 업무 종류, 난이도, 감정, 의도에 관한 분류
- 사용자의 콘텐츠 수집 활성·비활성 상태

본문 히스토리를 켜거나 꺼도 동일한 메타데이터를 입력한 활용 지수 결과는 같아야 한다. 콘텐츠 수집에 동의하지 않은 사용자를 점수나 신뢰도에서 불리하게 처리하지 않는다.

### 5.3 향후 분석

본문 또는 로컬 콘텐츠 분석을 도입하려면 이 정책의 단순 개정이 아니라 별도 정책, 별도 동의, 별도 위협 모델과 설계를 승인받아야 한다. LLM은 이미 계산된 집계 결과를 설명할 수는 있지만 점수나 세부 축을 결정·변경할 수 없다.

## 6. 공개 범위와 권한

### 6.1 개인 화면

- 개인 지수, 세부 축, 원시 비율, 기준 기간, 신뢰도는 해당 사용자 본인만 볼 수 있다.
- 관리자는 역할만으로 개인 지수 조회 권한을 얻지 않는다.
- 개인 지수 조회 권한은 UI뿐 아니라 서버 조회 경계에서 검사한다.

### 6.2 조직 화면

- 조직에는 개인별 행이 아닌 중앙값, 사분위 범위, 개인 기준선 대비 상태 비율 같은 익명 집계만 제공한다.
- 개인별 정렬, 검색, 드릴다운, CSV 내보내기, 식별 가능한 툴팁을 제공하지 않는다.
- 기존 비용·토큰 순위 데이터와 활용 지수 집계를 조인하거나 나란히 개인 단위로 비교하지 않는다.

### 6.3 최소 표본

- 선택한 기간과 필터를 적용한 활성 사용자가 5명 이상일 때만 집계를 반환한다.
- 5명 미만이면 숫자 `0`, 빈 배열, 이전 캐시 값을 반환하지 않고 `suppressed` 상태만 반환한다.
- 평균, 중앙값, 분포, 세부 축, 변화 비율을 모두 억제한다.
- 조직 전체, 팀, provider 등 모든 필터 조합에 같은 기준을 적용한다.
- 서로 다른 필터 결과를 빼서 개인을 추정하기 쉬운 세부 분할은 제공하지 않는다.

최소 표본 5명은 표본 억제 정책이며 수학적인 차등 개인정보 보호를 보장한다고 표현하지 않는다.

## 7. 데이터 부족과 신뢰도

- 데이터가 부족하면 낮은 점수를 부여하지 않고 `insufficient_data`로 표시한다.
- 지원되지 않는 provider 신호는 0으로 간주하지 않고 해당 세부 축에서 제외한다.
- `unknown` 도구 결과는 성공 또는 실패로 추정하지 않는다.
- 총점과 신뢰도는 별도 필드와 별도 시각 요소로 제공한다.
- 신규 사용자는 기준선이 준비되기 전까지 원시 관측값만 볼 수 있다.
- 데이터 누락, 낮은 결과 확인 비율, 일부 provider 미지원은 이유 코드와 함께 설명한다.

## 8. 투명성

모든 개인 지수에는 다음 정보를 함께 제공한다.

- 방법론 버전
- 현재 기간과 기준 기간
- 사용된 세부 축과 제외된 세부 축
- 각 세부 축의 원시 값과 계산 가능 여부
- 데이터 신뢰도와 근거
- 계산 시각과 데이터 최신성

점수는 `좋음`, `나쁨`, `상위`, `하위` 등 개인 평가 언어로 표시하지 않는다. `본인의 평소 범위`, `평소보다 높음`, `평소보다 낮음`처럼 자기 기준 표현만 사용한다.

## 9. 보존, 삭제, 정정

- 파생 지수는 원본 사용량·도구 이벤트보다 오래 보존하지 않는다.
- 원본이 삭제되거나 정정되면 캐시된 지수를 무효화하고 다음 조회에서 재계산한다.
- 첫 버전은 점수를 장기 저장하지 않고 요청 시 계산한 결과만 짧게 캐시한다.
- 사용자는 자신의 계산 근거와 데이터 범위를 확인하고 수집 오류를 신고할 수 있다.
- 특정 기기·기간의 원본 데이터가 잘못된 것으로 확인되면 원본 정정 또는 제외 후 재계산한다.

## 10. 보안 요구사항

- 개인 지수 조회는 현재 로그인 사용자 ID를 서버에서 강제한다.
- 조직 집계 함수는 개인별 점수 배열을 클라이언트에 반환하지 않는다.
- 표본 억제는 렌더링 이후가 아니라 집계 반환 전에 적용한다.
- 로그와 오류 메시지에 개인별 점수, 세부 축, 사용자 식별자 조합을 남기지 않는다.
- 캐시 키에는 사용자 또는 조직 범위, 기간, 방법론 버전, 필터를 포함한다.
- 개인 캐시와 조직 캐시를 공유하지 않는다.

## 11. 방법론 변경 관리

- 모든 산식에는 `utilization-vN` 형식의 버전을 부여한다.
- 가중치, 기준선, 최소 표본, 세부 축 정의가 바뀌면 방법론 버전을 올린다.
- 변경 이유, 적용일, 영향, 과거 데이터 재계산 여부를 방법론 변경 이력에 기록한다.
- 서로 다른 버전의 점수를 하나의 연속 추이로 표시하지 않는다.
- 과거 비교가 필요하면 동일 버전으로 재계산한 결과만 사용한다.

## 12. 출시 전 정책 점검

- 개인 지수를 관리자 권한으로 조회할 수 없는가?
- 5명 미만의 모든 필터 조합이 서버에서 억제되는가?
- 콘텐츠 수집 상태가 점수에 영향을 주지 않는가?
- 지원되지 않는 신호가 0점으로 처리되지 않는가?
- 점수와 신뢰도가 분리되어 있는가?
- 방법론 버전과 계산 근거가 화면에 표시되는가?
- 개인 평가, 순위, 생산성 단정 문구가 없는가?
- 삭제·정정 후 캐시가 무효화되는가?
