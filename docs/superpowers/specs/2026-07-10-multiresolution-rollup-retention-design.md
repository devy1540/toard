# 다중 해상도 Rollup·가격 이력·90일 원본 보존 설계

## 배경

toard의 사용량 대시보드는 원본 `usage_events`를 기준으로 정확한 토큰·비용·세션 집계를 제공해야 한다. 데이터가 누적돼도 최근 12개월 대시보드의 응답 속도를 예측 가능하게 유지하려면, 원본을 장기 조회 경로에서 분리하고 해상도별 집계를 사용해야 한다.

현재 ClickHouse 경로에는 원본과 시간별 rollup, 확정된 15분 rollup 및 raw tail을 합치는 경로가 있다. 그러나 시간별 rollup은 수집 배치에서 증분 합산하고, 15분 rollup만 watermark·dirty bucket·버전 재생성 규칙을 가진다. 원본 보존 기간, 늦게 도착한 이벤트의 허용 범위, 1일 rollup, 장기 조회의 상한은 코드로 확정돼 있지 않다.

가격도 같은 정합성 문제를 가진다. `pricing_models`는 `effective_date` 이력을 저장할 수 있지만, 수집 경로는 모델별 최신 가격만 로드해 이벤트 시각과 무관하게 비용을 확정한다. 따라서 모델 가격이 바뀐 뒤 과거 시각의 이벤트가 늦게 도착하면 잘못된 가격이 적용될 수 있다.

## 확정된 목표

- 원본 사용량 이벤트는 90일 보관한다.
- 수집 시각이 아니라 실제 사용 발생 시각(`UsageEvent.ts`) 기준으로 가격을 확정한다.
- 같은 모델의 가격이 날짜별로 바뀌어도 과거 이벤트 비용은 당시 가격으로 확정되고, 이후 가격 동기화가 과거 비용을 바꾸지 않는다.
- 90일 이내의 늦게 도착한 이벤트는 원본과 영향을 받는 rollup을 정확히 보정한다.
- 최근 12개월까지만 일반 대시보드 조회를 지원한다. 더 긴 기간은 일반 대시보드 범위에서 제외한다.
- 참조 환경의 캐시 미스 대시보드 응답은 P50 1초 이내, P95 2초 이내를 목표로 한다.
- 15분·시간·일별 rollup 결과는 같은 범위의 원본 집계와 동등해야 한다.

## 범위 제외

- 90일을 넘긴 원본 이벤트의 일반 수집 또는 자동 백필.
- 과거 전체 사용량을 새 가격표로 다시 계산하는 기능. 가격 변경은 앞으로 들어오는 이벤트에만 적용한다.
- 최근 12개월을 넘는 일반 대시보드, `period=all`의 무제한 집계, 무제한 커스텀 기간.
- 원본 사용량 본문·프롬프트 아카이브 또는 외부 오브젝트 스토리지 연동.
- 고객별 하드웨어·네트워크 조건까지 포함한 보편적 1초 응답 보장.

## 승인된 접근

원본은 90일 동안 정합성 기준으로 보존하고, 15분·시간·일별 rollup은 각각 원본에서 독립적으로 생성한다. `15분 → 시간 → 일`로 다시 합산하지 않는다. 연쇄 집계는 늦은 이벤트나 버그 보정 때 하위·상위 rollup을 모두 추적해야 하므로, 잘못된 중간 집계가 전파될 위험이 있다.

각 rollup은 자기 watermark와 dirty bucket을 가진 versioned 결과다. 읽기는 확정된 rollup 구간만 사용하고, 정렬되지 않은 요청 경계·watermark 뒤·dirty bucket은 원본으로 fallback한다. 모든 구간은 `[from, to)` 반열린 범위를 사용해 중복·누락을 막는다.

가격은 모델별 불변 이력에서 `event.ts` 이하인 가장 최근 유효 가격을 선택한다. 선택된 가격 revision과 계산 결과는 서버가 이벤트에 기록한다. 가격표 동기화가 과거 이벤트 또는 과거 rollup을 갱신하지 않는다.

## 데이터 수명과 조회 해상도

| 데이터 | 보관 기간 | 쓰임 | 정합성 규칙 |
| --- | --- | --- | --- |
| 원본 `usage_events` | 90일 | 지연 수집, 보정, rollup 재생성 | 90일 이내 이벤트의 유일한 재집계 기준 |
| 15분 rollup | 90일 | 당일·최근 기간의 15/30분 차트 | 30분 finalize 지연 뒤 원본에서 생성, dirty bucket 재생성 |
| 시간 rollup | 400일 | 과거 특정일의 시간 차트 | 원본에서 생성하는 versioned 결과 |
| 일별 rollup | 400일 | 최근 12개월 대시보드 | 원본에서 생성하는 versioned 결과 |
| 가격 이력 | 만료 없음 | 이벤트 발생 시각 기준 가격 선택 | 과거 revision은 변경·삭제하지 않음 |

400일은 최근 12개월 캘린더 기간, 윤년, 날짜 경계의 여유를 포함한다. 일반 대시보드는 최대 최근 12개월만 요청할 수 있다. 현재 `all` 프리셋과 12개월을 넘는 커스텀 기간은 UI와 서버 파서 모두에서 거부하거나 12개월로 제한한다. 단일 과거 날짜를 보는 경우에도 90일을 넘으면 15분 대신 시간 rollup을 사용한다.

원본 TTL과 허용 지연은 같은 90일이다. 서버는 `now - event.ts > 90일`인 이벤트를 집계에 넣지 않는다. 수집기는 이 응답을 재시도 실패로 해석하지 않도록 명시적인 `expired` 건수를 받으며, 서버는 운영 로그·메트릭에 해당 건수를 남긴다. 자동 백필은 하지 않는다.

## 가격 확정

### 가격 이력

가격 레코드는 다음 의미를 가진다.

- 정규화된 모델 식별자
- UTC `effective_at` 적용 시작 시각. 관리 화면에서 날짜만 입력하면 그 날짜 00:00 UTC로 정규화한다.
- 입력·출력·캐시·고속 모드·구간 가격
- 출처와 불변 revision 식별자

동일 모델의 새 가격은 기존 행을 덮어쓰지 않고 새 revision으로 추가한다. 공급자 가격 데이터에 실제 적용일이 없으면 최초 확인 시각부터 앞으로만 적용한다. 과거 적용일을 추정해 덮어쓰지 않는다. 관리자가 실제 적용일을 확인한 경우에만 새 revision으로 명시 등록할 수 있다.

가격 캐시는 모델별 최신 한 건이 아니라, 모델별 시간순 revision 전체를 보관한다. 서버는 `event.ts` 이하인 revision 중 `effective_at`이 가장 큰 한 건을 선택한다.

### 수집 시 처리

1. 서버는 클라이언트가 보낸 비용을 신뢰하지 않는다.
2. 이벤트의 `ts`를 기준으로 가장 최근 가격 revision을 찾는다.
3. 서버가 `cost_usd`, `pricing_revision_id`, `cost_status`를 기록한다.
4. `cost_status=priced`인 이벤트만 비용 합계에 포함한다.
5. 가격을 찾지 못한 이벤트는 `unpriced`로 남기고, 비용 0을 실제 무료 사용량으로 표시하지 않는다.

`pricing_revision_id`, `priced_event_count`, `unpriced_event_count`는 원본과 모든 rollup의 그룹 키 또는 집계 필드에 포함한다. 따라서 원본이 만료된 뒤에도 비용 합계의 가격 확정 상태를 설명할 수 있다. 새 가격 revision은 이미 저장된 이벤트와 rollup을 수정하지 않는다.

## Rollup 생성과 보정

### 원본 기반 독립 생성

15분·시간·일별 compactor는 각각 해당 버킷의 원본 이벤트를 그룹화해 새 version을 쓴다. 시간별 현재 `SummingMergeTree` 증분 쓰기는 versioned 원본 재생성 방식으로 전환한다. 이로써 늦은 이벤트, 가격 선택 버그의 90일 내 수정, 재시도 중복을 같은 규칙으로 처리한다.

각 결과는 최소한 다음 차원을 유지한다.

- 버킷 시작 시각(UTC)
- provider, user, team, session, model, host
- 가격 revision과 가격 확정 상태
- 이벤트 수, 입력·출력·캐시 토큰, 비용

`ReplacingMergeTree(version)` 조회는 최신 version을 `argMax`로 선택한다. 물리 병합 완료 여부에 따라 결과가 달라지지 않도록 대시보드 쿼리에서 명시적으로 최신값을 고른다.

### Watermark와 dirty bucket

원본 이벤트가 성공적으로 저장되면 이벤트가 속한 15분·시간·일별 버킷을 모두 dirty로 표시한다. 각 compactor는 다음 순서로 동작한다.

1. 자기 finalize 지연을 넘긴 dirty bucket과 watermark 앞의 미처리 버킷을 잠근다.
2. 선택한 버킷을 원본에서 다시 집계해 새 version으로 쓴다.
3. 쓰기 성공 뒤에만 watermark를 전진시키고 dirty 표식을 제거한다.
4. 실패하면 watermark와 dirty 표식을 유지해 다음 tick에서 재시도한다.

원본이 남아 있는 90일 동안만 재생성이 가능하다. 90일 이전 이벤트는 수집 경로에서 집계 제외하므로, 원본 TTL 뒤 기존 rollup을 불완전하게 재생성하는 경우가 없다.

## 조회 라우팅

저장소는 요청 버킷과 기간을 기준으로 가장 거친 정확한 소스를 선택한다.

| 요청 | 기본 소스 | 예외 |
| --- | --- | --- |
| 오늘의 15/30분 | 15분 rollup | 확정 전·dirty·부분 버킷은 원본 |
| 특정 날짜의 시간 | 시간 rollup | 확정 전·dirty·부분 버킷은 원본 |
| 2일 이상, 최대 12개월 | 일별 rollup | 오늘의 미확정 부분은 원본 또는 시간 rollup |

source를 바꾸는 경계는 watermark와 요청의 정렬된 버킷 경계를 함께 만족해야 한다. 한 요청 안에서도 rollup 구간과 raw tail을 `UNION ALL`할 수 있지만, 두 소스가 같은 이벤트 시간 범위를 포함해서는 안 된다. 90일을 넘긴 원본 테이블은 어떤 일반 대시보드 쿼리에도 사용하지 않는다.

## 성능 기준

성능 기준은 네트워크 왕복을 제외하고, 인증된 대시보드 HTTP 요청이 서버에 들어온 뒤 응답 본문을 완료할 때까지의 시간이다. 응답 캐시는 비운 상태에서 측정한다.

- 대상: ClickHouse 백엔드, 400일에 걸친 100만 건의 사용 이벤트, 사용자 100명·provider 5개·모델 10개를 포함한 고정 벤치마크 데이터셋.
- 환경: Docker Compose에서 앱·Postgres·ClickHouse에 총 4 vCPU와 8 GiB 메모리를 할당한 참조 환경.
- 시나리오: 전체 조직의 최근 12개월 일별 대시보드, 모델·provider·팀 필터, 개별 사용자의 최근 12개월 대시보드.
- 판정: 각 시나리오를 100회 실행해 P50 1초 이내, P95 2초 이내.

이는 참조 환경의 릴리스 기준이다. 모든 셀프호스팅 머신의 네트워크와 자원을 포함한 절대 보장은 아니다. 참조 환경에서 기준을 넘기면 더 큰 원본 TTL이나 더 긴 조회 범위를 추가하지 않고, 쿼리 계획·정렬 키·캐시를 먼저 개선한다.

## 전환과 롤백

1. 새 스키마와 compactor를 배포하되 읽기 플래그는 끈다.
2. 15분·시간·일별 rollup을 shadow로 생성한다.
3. 모든 지원 범위와 필터에서 원본 대 rollup 결과가 동일한지 검증한다.
4. 일별, 시간, 15분 순으로 읽기를 독립 플래그로 전환한다.
5. 각 전환 뒤 `/api/ready`, compactor watermark, dirty bucket 수, 앱·ClickHouse 오류 로그를 관찰한다.

불일치·watermark 정지·성능 기준 초과가 발생하면 해당 해상도의 읽기 플래그만 끄고 raw 또는 더 세밀한 검증 완료 rollup으로 되돌린다. 원본 TTL은 세 해상도의 shadow 검증과 읽기 전환이 완료되기 전에는 켜지지 않는다.

## 테스트 기준

### 정합성

- 같은 모델의 가격이 바뀐 전후 이벤트와, 가격 변경 전 시각에 늦게 도착한 이벤트가 각각 올바른 revision으로 비용을 확정한다.
- 가격이 없는 모델은 비용 0으로 보이지 않고 `unpriced`로 집계된다.
- 원본과 15분·시간·일별 결과가 토큰, 비용, 이벤트 수, 세션 수, 활성 사용자 수에서 동일하다.
- provider·user·team·model·host 필터와 부분 버킷, 뷰어 타임존, 12개월 경계에서도 동일하다.
- 90일 이내 늦은 이벤트는 모든 영향 버킷의 version을 올려 원본과 동일해진다.
- 90일 초과 이벤트는 집계에 반영되지 않고 `expired`로 관측된다.
- 재시도·중복 이벤트는 원본과 모든 rollup에서 한 번만 계산된다.

### 성능과 운영

- 위 성능 기준의 각 시나리오가 P50 1초, P95 2초를 통과한다.
- raw tail, dirty fallback, 완전 rollup 조회를 각각 포함한 통합 테스트가 통과한다.
- watermark가 단조 증가하고 compactor 실패 뒤 재시도된다.
- 읽기 플래그를 끄면 검증된 이전 소스로 즉시 복귀한다.
- `pnpm -r typecheck`, `pnpm -r test`, 정확성 검증 스크립트, 벤치마크, `git diff --check`가 통과한다.

## 완료 조건

- 최근 12개월 대시보드는 원본 전체 스캔 없이 일별 rollup을 사용한다.
- 최근 90일 원본 보정과 그보다 오래된 rollup 보존의 경계가 코드·운영 지표·사용자 응답에 명시돼 있다.
- 날짜별 모델 가격 변경과 늦은 이벤트가 이벤트 시각 기준으로 올바르게 계산된다.
- 장기 재가격 계산 없이도 이미 확정된 비용과 rollup은 변하지 않는다.
- 참조 환경에서 정의한 P50/P95 기준과 모든 원본 대 rollup 동등성 검증을 통과한다.
