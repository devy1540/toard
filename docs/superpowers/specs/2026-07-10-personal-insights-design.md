# 개인 인사이트 설계

## 배경

toard의 개인 영역은 현재 `내 사용량`과 `히스토리`를 제공한다. 내 사용량 화면은 비용·세션·토큰·모델·도구·기기 집계를 보여주지만, 사용자가 직전 기간과 비교해 무엇이 달라졌는지를 직접 해석해야 한다.

개인 영역에 `인사이트` 메뉴를 추가해 현재 기간과 직전 기간의 사용 패턴을 비교하고, 집계 결과에서 의미 있는 변화만 규칙 기반 문장으로 요약한다. 첫 버전은 별도 모델을 호출하지 않는다. 이후 AI 해석을 도입할 때도 동일한 집계 결과를 입력으로 재사용할 수 있도록 데이터 조회와 해석을 분리한다.

## 목표

- 개인 메뉴에 `인사이트`를 추가하고 `/insights` 화면을 제공한다.
- 최근 7일, 이번 주, 이번 달의 비용·세션·토큰 변화를 직전 대응 기간과 비교한다.
- 모델과 도구의 사용 비중 변화를 보여준다.
- 의미 있는 변화만 최대 3개의 규칙 기반 문장으로 요약한다.
- ClickHouse 조회는 15분 롤업과 최근 raw tail을 활용하고 캐시 미스당 쿼리 수를 제한한다.
- 사용자별 결과를 10분간 캐시해 반복 방문이 ClickHouse와 서버에 주는 부담을 줄인다.
- PostgreSQL과 ClickHouse 백엔드가 동일한 인사이트 데이터 계약을 제공한다.

## 범위 제외

- LLM을 호출하는 AI 해석과 추천.
- 인사이트 결과의 장기 저장, 일별 스냅샷 테이블, 알림 발송.
- 직접 선택 기간, 분기·연간·전체 기간 비교.
- 팀·조직 인사이트.
- 사용자의 생산성이나 업무 품질에 대한 판단.
- 모델 가격 최적화처럼 사용 행동을 지시하는 추천.

## 메뉴와 라우트

개인 메뉴 순서는 다음과 같다.

1. 내 사용량 (`/`)
2. 인사이트 (`/insights`)
3. 히스토리 (`/history`)

인사이트 아이콘은 기존 Lucide 아이콘 집합에서 의미가 명확한 `Lightbulb`를 사용한다. 메뉴명과 화면 문자열은 `nav`, `insights` 한국어·영어 메시지 카탈로그에 추가한다. `/insights`와 하위 경로는 기존 사이드바 prefix 활성 규칙을 따른다.

## 기간과 비교 의미

첫 버전은 다음 세 프리셋만 제공한다.

| 프리셋 | 현재 기간 | 이전 기간 |
|---|---|---|
| 최근 7일 | 현재 시각까지의 최근 7일 | 그 직전 7일 |
| 이번 주 | 뷰어 타임존의 이번 주 시작부터 현재 시각 | 지난주 같은 경과 시점까지 |
| 이번 달 | 뷰어 타임존의 이번 달 1일부터 현재 시각 | 지난달 같은 경과 시점까지 |

이번 주와 이번 달은 진행 중인 현재 기간을 완료된 이전 주·월 전체와 비교하지 않는다. 같은 경과 길이를 비교해 월초나 주초의 변화율이 과장되지 않게 한다. 지난달에 같은 일자가 없으면 지난달 말일로 상한을 맞춘다.

`provider` 필터는 기존 내 사용량 화면과 동일하게 적용한다. 첫 버전에는 직접 선택을 두지 않아 현재 기간과 이전 기간을 합쳐도 조회 범위가 두 달을 넘지 않는다.

## 화면 구성

승인된 `요약 우선` 레이아웃을 사용한다.

1. 헤더
   - 제목 `인사이트`.
   - 기간 프리셋 선택.
   - 비교 대상 기간 라벨.
   - 결과 계산 시각과 `최대 10분 전 데이터` 안내.
2. 핵심 변화
   - 우선순위가 높은 규칙 기반 인사이트 1~3개.
   - 대표 문장과 짧은 근거 문장.
3. 지표 카드
   - 비용, 세션, 전체 토큰.
   - 현재 값과 이전 기간 대비 증감.
   - 임계값 미만의 작은 변화도 실제 수치로 표시.
4. 기간별 변화
   - 현재와 이전 기간을 같은 경과 위치에 맞춘 추이 차트.
   - 기본 지표는 비용이며 비용·토큰 전환을 제공한다.
5. 구성 변화
   - 모델과 도구 탭.
   - 현재 비중, 이전 비중, 퍼센트포인트 차이.
   - 가장 큰 변화부터 정렬한다.

화면은 기존 대시보드의 카드, 필터, 차트, 빈 상태 컴포넌트를 재사용한다. 별도 디자인 시스템이나 새로운 차트 라이브러리를 추가하지 않는다.

## 데이터 계약

`@toard/core`에 백엔드 공통 반환 형식을 추가한다.

```ts
interface InsightMetricSummary {
  totalCostUsd: number;
  totalSessions: number;
  totalTokens: number;
}

interface InsightTrendPoint {
  position: number;
  current: { costUsd: number; sessions: number; totalTokens: number };
  previous: { costUsd: number; sessions: number; totalTokens: number };
}

interface InsightCompositionChange {
  key: string;
  currentCostUsd: number;
  previousCostUsd: number;
  currentTokens: number;
  previousTokens: number;
}

interface UserInsightComparison {
  current: InsightMetricSummary;
  previous: InsightMetricSummary;
  trend: InsightTrendPoint[];
  byModel: InsightCompositionChange[];
  byProvider: InsightCompositionChange[];
}
```

실제 타입명과 필드 배치는 구현 중 기존 core 명명 규칙에 맞춰 조정할 수 있지만, UI와 규칙 엔진은 저장소별 SQL이나 raw 이벤트 형식을 알지 못해야 한다.

`StorageBackend`에는 사용자와 현재·이전 기간을 받아 `UserInsightComparison`을 반환하는 전용 메서드를 추가한다. 내 사용량의 `getUserUsage()`를 두 번 호출하지 않는다.

## ClickHouse 조회

### 15분 롤업 사용

현재 ClickHouse 구현은 `usage_15m_rollup`과 raw 데이터를 합치는 하이브리드 source를 제공한다. 인사이트 집계도 이 source를 사용한다.

- 확정된 중간 구간: `usage_15m_rollup`.
- 부분 버킷과 최신 구간: `usage_events` raw.
- watermark가 없거나 범위 안에 dirty bucket이 있으면 해당 구간부터 raw로 폴백.
- `CLICKHOUSE_READ_15M_ROLLUP`이 꺼져 있으면 정확성을 유지하기 위해 raw source 사용.

현재 하이브리드 helper는 시계열 조회에만 사용되므로, 인사이트 구현에서는 요약과 구성 집계도 동일한 source를 사용할 수 있게 내부 경계를 정리한다. 기존 시간별 롤업을 인사이트의 기본 source로 새로 선택하지 않는다.

### 쿼리 예산

캐시 미스 한 번당 ClickHouse 쿼리 요청은 최대 2개로 제한한다.

1. 현재·이전 기간의 요약과 위치 정렬 추이.
2. 현재·이전 기간의 모델·도구 구성.

두 기간을 각각 조회하지 않고, 연속된 전체 범위를 한 번 읽어 `current`와 `previous`로 분류한다. 쿼리 결과 행 수는 프리셋의 표시 버킷과 구성 차원으로 제한하고 raw 이벤트를 애플리케이션으로 전달하지 않는다.

구현 후 ClickHouse query log와 raw/15분 하이브리드 동등성 검증으로 실제 스캔량과 결과를 확인한다. 쿼리 예산을 맞출 수 없는 SQL 구조가 확인되면 기능을 늘리지 않고 구성 차트를 축소한다.

## PostgreSQL 조회

PostgreSQL 구현도 같은 `StorageBackend` 메서드를 제공한다. `usage_events`를 현재·이전 전체 범위로 제한하고 SQL 집계 결과만 반환한다. 프리셋이 최대 이번 달이므로 직접 선택이나 전체 기간 조회는 발생하지 않는다.

PostgreSQL과 ClickHouse 구현은 다음 값이 같아야 한다.

- 비용, 세션, 전체 토큰 합계.
- 표시 버킷별 비용·세션·토큰.
- 모델·도구별 비용과 토큰.
- 빈 모델·도구 값의 정규화 방식.

## 10분 결과 캐시

캐시는 15분 롤업과 목적이 다르다. 15분은 저장 집계 단위이고, 10분은 동일한 인사이트 요청 결과를 재사용하는 TTL이다.

- 키: 사용자 ID, 현재 기간 시작·종료, 이전 기간 시작·종료, provider 필터, 뷰어 타임존.
- TTL: 10분.
- 범위: 집계 DTO까지만 캐시하고 번역된 문장이나 React 결과는 캐시하지 않는다.
- 사용자 ID가 반드시 키에 포함되어 사용자 간 결과가 섞이지 않게 한다.
- 기간이나 provider가 달라지면 별도 키를 사용한다.
- 자동 새로고침은 제공하지 않는다.
- 캐시가 만료된 다음 요청이 새 집계를 수행한다.

UI에는 계산 시각을 표시해 최대 10분의 지연 가능성을 숨기지 않는다. 첫 버전에는 강제 새로고침 버튼과 별도 캐시 테이블을 추가하지 않는다.

## 규칙 엔진

규칙 엔진은 `UserInsightComparison`만 입력받는 순수 함수로 구현한다. 반환값은 번역 가능한 인사이트 키, 변수, 중요도, 근거 지표다. 완성된 한국어·영어 문장을 데이터 계층에서 만들지 않는다.

### 후보 규칙

- 비용 증감.
- 세션 증감.
- 전체 토큰 증감.
- 세션당 비용 증감.
- 모델별 비용 또는 토큰 비중 변화.
- 도구별 비용 또는 토큰 비중 변화.

### 노출 기준

- 비용·세션·토큰·세션당 비용: 절대 변화율 10% 이상.
- 모델·도구 구성: 절대 점유율 변화 5%p 이상.
- 비율 기반 문장은 양쪽 기간 모두 세션 5개 이상일 때 생성.
- 이전 값이 0이고 현재 값이 양수면 무한대 변화율을 만들지 않고 `새롭게 사용` 후보를 생성.
- 현재와 이전 값이 모두 0이면 후보를 만들지 않음.
- 중요도와 변화 크기로 정렬해 최대 3개 노출.

임계값은 문장 노출만 제어한다. KPI 카드와 차트는 임계값에 상관없이 실제 값을 표시한다.

### 표현 원칙

- 관측된 수치만 설명하고 생산성이나 업무 품질을 추정하지 않는다.
- `좋아졌다`, `나빠졌다`는 세션당 비용처럼 근거가 명확한 효율 지표에만 제한한다.
- 데이터가 적을 때는 단정적인 문장을 만들지 않는다.
- 향후 AI 해석기는 규칙 엔진과 같은 DTO를 입력받는 별도 구현으로 추가한다.

## 빈 상태와 오류 처리

- 로그인 사용자 ID가 없으면 기존 개인 대시보드와 같은 로그인 필요 상태를 표시한다.
- 현재 기간 데이터가 없으면 설치 상태와 기간 변경 안내를 제공한다.
- 이전 기간 데이터가 없으면 증감률을 숨기고 신규 사용 문구 또는 현재 절대값만 표시한다.
- 모델이나 도구 구성이 없으면 해당 섹션만 빈 상태로 표시한다.
- rollup watermark가 없거나 dirty bucket이 있으면 기존 하이브리드 정책에 따라 raw로 폴백한다.
- 저장소 조회 실패 시 임의의 0값이나 인사이트를 만들지 않고 기존 대시보드 오류 경계로 전달한다.
- 캐시 계층은 원본 오류를 성공 결과로 저장하지 않는다.

## 테스트 기준

### 기간

- 최근 7일과 직전 7일이 연속되고 겹치지 않는다.
- 이번 주와 지난주가 뷰어 타임존에서 같은 경과 길이를 가진다.
- 이번 달과 지난달이 월 길이가 다를 때 유효한 말일로 맞춰진다.
- DST 전환이 있는 타임존에서도 캘린더 경계가 올바르다.

### 규칙

- 10% 미만 변화는 KPI에는 보이지만 문장 후보에서 제외된다.
- 정확히 10%인 변화는 후보가 된다.
- 5%p 미만 구성 변화는 제외되고 정확히 5%p는 포함된다.
- 세션 5개 미만에서는 비율 기반 문장을 만들지 않는다.
- 이전 값 0, 양쪽 값 0, 음수 방향, 동률을 각각 검증한다.
- 후보가 3개를 넘으면 중요도 순으로 3개만 남는다.

### 저장소

- 동일 fixture에서 PostgreSQL, ClickHouse raw, ClickHouse 15분 하이브리드 결과가 일치한다.
- partial boundary와 최신 raw tail을 포함해도 중복 합산되지 않는다.
- dirty bucket이 있으면 정확한 raw 결과로 폴백한다.
- 캐시 미스당 ClickHouse 쿼리 요청이 2개를 넘지 않는다.

### UI

- 개인 메뉴 순서와 `/insights` 활성 상태가 맞다.
- 세 프리셋이 URL과 화면 상태에 반영된다.
- 한국어·영어에서 모든 문장과 변수 치환이 정상이다.
- 데스크톱, 접힌 사이드바, 모바일 드로어에서 메뉴와 화면이 정상이다.
- 승인된 `요약 우선` 목업과 실제 화면의 정보 계층을 브라우저에서 비교한다.

### 완료 검증

- 관련 단위·저장소 테스트.
- `pnpm --filter @toard/web typecheck`.
- `git diff --check`.
- 로컬 `/insights` 응답과 실제 브라우저 렌더 확인.
- ClickHouse query log에서 인사이트 캐시 미스의 쿼리 요청 수와 source를 확인.

## 점진적 확장

첫 버전이 안정화된 뒤 다음 순서로 확장할 수 있다.

1. query log 근거를 바탕으로 직접 선택 기간 추가.
2. 규칙 후보와 근거 지표에 대한 사용자 피드백 수집.
3. 같은 `UserInsightComparison`을 입력받는 AI 해석기 추가.
4. AI 결과 캐시·비용 한도·개인정보 정책을 별도 설계.

AI 해석을 추가하더라도 규칙 기반 결과는 폴백과 수치 근거로 유지한다.
