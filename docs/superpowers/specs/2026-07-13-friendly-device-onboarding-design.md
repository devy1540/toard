# 쉬운 기기 연결 온보딩 설계

## 배경

현재 `설정 → 설치 · 토큰`은 토큰 발급, 토큰 라벨, 본문 수집 설정, 설치 명령, 고급 설정, 제거, 토큰 관리, 기기 상태를 한 화면에 함께 보여준다. 개발자는 명령을 해석할 수 있지만, 비개발자는 어떤 터미널을 열어야 하는지와 설치가 끝났는지를 스스로 판단해야 한다.

Windows x64 shim 바이너리는 GitHub Release에 게시되지만 현재 온보딩은 macOS·Linux용 `sh` 명령만 생성한다. README가 안내하는 `npx @toard/shim` 패키지는 npm 게시 상태에 의존하므로 Windows의 기본 설치 경로로 삼지 않는다.

## 목표

- 개발 경험과 관계없이 한 번에 한 행동만 따라 하면 설치를 끝낼 수 있게 한다.
- 브라우저에서 Windows, macOS, Linux를 감지하고 해당 OS 명령만 기본 표시한다.
- 사용자가 토큰 이름이나 인증 구조를 이해하지 않아도 `이 컴퓨터 연결하기`로 토큰을 자동 발급한다.
- Windows는 PowerShell 한 줄 설치, macOS·Linux는 기존 셸 한 줄 설치를 제공한다.
- 설치 명령 실행 뒤 첫 인증 요청을 자동 감지해 연결 완료를 알려준다.
- 기존 토큰 관리, 제거, 수동 설정은 유지하되 초기 설치 흐름과 분리한다.
- 현재의 본문 수집 정책과 사용자 선택을 그대로 보존한다.

## 범위 제외

- 서명된 `.exe`, `.msi`, `.pkg` 또는 GUI 설치 앱 배포.
- Windows 예약 작업을 이용한 주기 수집 데몬 등록.
- 브라우저에서 로컬 터미널이나 설치 프로그램을 직접 실행하는 기능.
- npm 패키지 게시를 Windows 설치의 필수 조건으로 만드는 작업.
- 토큰·기기 데이터 모델의 전면 개편.

## 사용자 흐름

초기 설치는 `이 컴퓨터 연결하기` 진입 화면과 3단계 마법사로 구성한다.

### 진입 화면

- 제목: `이 컴퓨터를 toard에 연결하세요`.
- 사용량이 내 계정에 기록된다는 결과를 먼저 설명한다.
- 현재 수집 범위를 한 문장으로 표시한다.
  - 본문 수집이 꺼져 있으면 프롬프트와 코드 내용은 보내지 않는다고 표시한다.
  - 본문 수집 기능이 켜져 있으면 현재 기본값과 토글을 명시적으로 보여준다.
- 기본 동작은 `이 컴퓨터 연결하기` 하나다.
- 토큰 라벨 입력, 평문 토큰 상태, 설치 명령, 제거 기능은 진입 화면에서 숨긴다.

### 1단계: 운영체제 확인

- 클라이언트에서 운영체제를 감지해 `Windows`, `macOS`, `Linux` 중 하나를 기본 선택한다.
- 감지 우선순위는 `navigator.userAgentData.platform`, `navigator.platform`, `navigator.userAgent` 순서다.
- 감지하지 못하면 선택을 강제하지 않고 세 OS 버튼을 보여준다.
- 자동 감지는 편의 기능이며 사용자가 언제든 선택을 바꿀 수 있다.
- 지원 범위는 Windows x64, macOS arm64/x64, Linux arm64/x64다. Windows arm64는 지원 대상으로 표시하지 않는다.
- `계속`을 누르는 시점에 이 컴퓨터용 토큰을 자동 발급한다. OS 확인 전에 토큰을 만들지 않아 중도 이탈로 생기는 미사용 토큰을 줄인다.

### 2단계: 설치 명령 실행

- 선택한 OS에 맞는 안내만 표시한다.
- Windows는 시작 메뉴에서 PowerShell을 여는 법, macOS는 Spotlight에서 터미널을 여는 법, Linux는 사용 중인 터미널을 여는 법을 짧게 설명한다.
- 사용자는 `설치 명령 복사`를 누르고 터미널에 붙여넣은 뒤 Enter를 누른다.
- 명령에는 방금 발급한 토큰과 현재 endpoint, 본문 수집 선택이 반영된다.
- 평문 토큰은 명령 안에 포함되지만 별도 토큰 필드나 설명으로 반복 노출하지 않는다. 화면 이탈 후에는 다시 조회할 수 없다.
- `명령을 실행했어요`를 누르면 3단계로 이동한다.

플랫폼별 명령 형태는 다음과 같다.

```powershell
$env:TOARD_INGEST_TOKEN='<token>'; $env:TOARD_SHIM_COLLECT_CONTENT='<0|1>'; irm '<baseUrl>/install.ps1' | iex
```

```sh
curl -fsSL '<baseUrl>/install.sh' | TOARD_INGEST_TOKEN='<token>' TOARD_SHIM_COLLECT_CONTENT='<0|1>' sh
```

실제 생성 함수는 PowerShell 작은따옴표와 POSIX 셸 작은따옴표를 안전하게 이스케이프한다. `baseUrl`과 endpoint는 서버가 생성한 공용 URL만 사용한다.

### 3단계: 연결 자동 확인

- 설치 화면에서 발급한 `tokenId`를 기준으로 현재 로그인 사용자 소유 토큰의 `last_used_at`을 2초마다 확인한다.
- 최대 2분 동안 확인하고, 연결되면 즉시 성공 화면으로 전환한다.
- 설치 스크립트 마지막의 `toard-shim doctor`가 빈 인증 요청을 보내므로 사용 로그가 없는 새 컴퓨터도 `last_used_at`이 갱신된다.
- doctor의 빈 요청에는 host 정보가 없으므로 성공 화면은 `Windows 컴퓨터 연결됨`처럼 OS를 표시한다. 실제 컴퓨터 이름은 첫 사용량 수집 후 기기 관리 영역에 표시한다.
- 성공 화면의 기본 동작은 `내 사용량 보러 가기`, 보조 동작은 `다른 컴퓨터도 연결하기`다.

### 확인 지연과 오류 복구

- 2분 안에 신호가 없으면 원시 오류나 토큰을 노출하지 않는다.
- 다음 순서로 복구 행동을 보여준다.
  1. 터미널에 `완료` 또는 오류가 표시됐는지 확인한다.
  2. 같은 설치 명령을 다시 복사해 실행한다. 설치기는 멱등이어야 한다.
  3. `toard-shim doctor` 명령을 복사해 진단한다.
  4. 운영체제 선택으로 돌아간다.
- 사용자가 마법사를 닫아도 새로 발급된 토큰은 토큰 관리에서 폐기할 수 있다.

## Windows PowerShell 설치기

`/install.ps1`을 동적 라우트로 제공한다. 설치기는 npm을 거치지 않고 최신 GitHub Release 자산을 직접 사용한다.

1. `x86_64-pc-windows-msvc.exe`와 `SHA256SUMS`를 임시 디렉터리에 다운로드한다.
2. `Get-FileHash -Algorithm SHA256` 결과와 게시된 checksum을 비교한다.
3. 불일치하면 설치를 중단하고 자격 증명이나 PATH를 수정하지 않는다.
4. `%USERPROFILE%\.toard\bin`에 `claude.exe`, `codex.exe`, `toard-shim.exe` 사본을 설치한다.
5. `%USERPROFILE%\.toard\credentials`를 BOM 없는 UTF-8로 작성한다.
6. 사용자 PATH 앞에 shim 디렉터리를 멱등으로 추가하고 현재 PowerShell 세션 PATH에도 반영한다.
7. `toard-shim doctor`를 실행해 endpoint 인증과 PATH 상태를 확인한다.
8. 성공 시 비기술적인 `toard 연결 완료` 메시지를 출력한다.

비밀값과 인증 헤더는 출력하지 않는다. checksum 오류, 다운로드 실패, 파일 교체 실패는 해당 단계에서 중단하고 사용자에게 다시 실행 가능한 짧은 메시지를 보여준다.

Windows 관리 영역에는 `/uninstall.ps1` 기반 제거 명령도 제공한다. 제거기는 `.toard/bin`의 toard 소유 사본과 credentials, 사용자 PATH의 toard 항목만 제거하며 원래 설치된 Claude·Codex에는 손대지 않는다. macOS·Linux는 기존 `/uninstall.sh`를 계속 사용한다.

## 컴포넌트와 데이터 흐름

### 클라이언트 마법사

기존 `OnboardingPanel`의 책임을 다음 단위로 나눈다.

- 플랫폼 감지와 명령 생성: 부작용 없는 함수로 분리해 Node 테스트에서 검증한다.
- 연결 마법사: 단계, 선택 OS, 발급 결과, 확인 상태를 관리한다.
- 연결 관리: 기존 토큰 목록, 기기 목록, 제거·고급 설정을 초기 마법사 아래에서 유지한다.

마법사 상태는 다음 값으로 제한한다.

```text
intro → platform → install → verifying → success
                                  └────→ stalled
```

새로고침 후 평문 토큰은 복구할 수 없으므로 진행 중 마법사를 영구 저장하지 않는다. 새로고침한 사용자는 새 연결을 시작하거나 미사용 토큰을 관리 화면에서 폐기한다.

### 토큰 발급과 상태 확인

- 기존 `issueToken()`의 문자열 반환 계약은 유지한다.
- 마법사용 별도 함수는 `{ token, tokenId }`를 반환하도록 DB `INSERT ... RETURNING id`를 사용한다.
- 서버 액션은 로그인 사용자 ID를 강제하고 `{ token, tokenId }`를 클라이언트에 한 번만 반환한다.
- 상태 확인 서버 액션은 `user_id`와 `tokenId`를 함께 조건으로 조회하며 다른 사용자의 토큰 상태를 반환하지 않는다.
- 확인 응답은 `connected`, `lastUsedAt`, `lastHost`만 제공하고 token hash나 평문 토큰을 포함하지 않는다.

### 기존 관리 영역

- `토큰 관리`와 `내 기기 · 수신 확인`은 설치 마법사 아래에 유지한다.
- `직접 설정`, `제거`, 토큰 폐기는 `연결된 컴퓨터 관리` 또는 각 기기 작업 메뉴에서 접근한다.
- 이미 연결된 기기가 있어도 새 컴퓨터 연결 CTA는 항상 제공한다.
- 기존 활성 토큰은 새 토큰 발급 시 폐기하지 않는다.

## 다국어와 접근성

- 한국어와 영어 메시지를 동시에 추가한다.
- 단계 진행은 색상뿐 아니라 `1/3`, 단계 제목, 완료 문구로 표현한다.
- OS 선택은 실제 `button` 또는 radio semantics를 사용하고 키보드로 조작할 수 있게 한다.
- 복사 성공은 toast뿐 아니라 버튼 라벨 변경 또는 인접 상태 문구로 알린다.
- 320px 폭에서 명령 영역과 버튼이 가로로 넘치지 않게 줄바꿈·스크롤 정책을 검증한다.
- `prefers-reduced-motion` 사용자는 대기 애니메이션 없이 상태 문구만 갱신한다.

## 테스트와 검증

- 플랫폼 감지 함수가 Windows, macOS, Linux, unknown을 정확히 반환하는지 단위 테스트한다.
- Windows PowerShell과 POSIX 셸 명령 생성이 토큰, endpoint, 본문 수집 값을 정확히 이스케이프하는지 테스트한다.
- 마법사용 토큰 발급이 token ID를 반환하면서 기존 활성 토큰을 폐기하지 않는지 테스트한다.
- 상태 확인이 로그인 사용자 소유 토큰만 조회하는지 테스트한다.
- `/install.ps1` 응답의 필수 다운로드, SHA256 검증, BOM 없는 credentials 작성, PATH 멱등 처리 구문을 테스트한다.
- `/uninstall.ps1`이 toard 소유 파일과 PATH 항목만 제거하고 원래 Claude·Codex 경로를 건드리지 않는지 테스트한다.
- 기존 `/install.sh`와 Windows 설치기 모두 doctor를 실행하는지 테스트한다.
- 한국어·영어 메시지 키가 일치하는지 검증한다.
- web 테스트, typecheck, shim Rust 테스트, `git diff --check`를 실행한다.
- 로컬 설정 화면에서 Windows, macOS, Linux 선택 상태와 320px 좁은 폭을 렌더링해 확인한다.
- 실제 Windows PowerShell 검증이 불가능한 환경에서는 이를 명시하고, GitHub Actions `windows-latest`에서 PowerShell 구문 분석과 설치기 테스트를 수행한다.

## 완료 기준

- 사용자는 토큰 용어를 이해하지 않아도 새 컴퓨터 연결을 시작할 수 있다.
- Windows, macOS, Linux에서 선택한 OS에 맞는 한 개의 설치 명령만 보인다.
- Windows 명령이 Unix 셸 문법을 포함하지 않는다.
- 설치 명령 실행 후 별도 사용량 이벤트가 없어도 2분 안에 연결 완료 상태가 표시된다.
- 수신 지연 시 다시 실행, doctor, OS 재선택 경로가 제공된다.
- 기존 토큰 폐기, 기기 상태, 수동 설정, 제거 기능이 사라지지 않는다.
- 비밀값은 설치 명령의 일회성 표시 외에 로그, 오류, 관리 목록에 노출되지 않는다.
