# E2EE 다중 기기 통합 연결 설계

- 상태: 방향 승인 완료, 작성 명세 사용자 검토 대기
- 작성일: 2026-07-17
- 대상: 컴퓨터 연결 온보딩, shim E2EE 키 등록, 브라우저 히스토리 잠금 해제, 승인 기기 관리
- 선행 설계: `2026-07-14-e2ee-prompt-history-design.md`

## 1. 결정 요약

사용자에게 수집 기기와 열람 브라우저를 별개 제품 개념으로 노출하지 않는다. 사용자는 새 컴퓨터에서 **이 컴퓨터 연결**을 한 번 수행하고, 기존 신뢰 기기 또는 Recovery Kit로 한 번 승인한다.

연결이 완료되면 새 컴퓨터는 다음 두 능력을 함께 얻는다.

- 새 컴퓨터의 shim이 로컬 프롬프트·응답을 기존 사용자 E2EE 키로 암호화해 수집한다.
- 새 컴퓨터의 현재 브라우저가 같은 사용자의 기존 전체 히스토리를 복호화해 열람한다.

히스토리는 컴퓨터별 소유물이 아니라 사용자 계정의 E2EE 히스토리다. 맥북에서 수집한 기록과 맥미니에서 수집한 기록은 같은 `content_owner_id`와 UCK 버전에 속하며, 승인된 어느 기기에서든 전체를 볼 수 있다.

서버는 계속 평문 UCK를 알지 못한다. 새 컴퓨터는 기존 신뢰 기기가 새 shim 키와 브라우저 키 각각에 암호화한 UCK envelope를 받는다.

## 2. 현재 문제

현재 구현은 최초 E2EE 설정과 새 브라우저 승인만 지원한다.

- `toard-shim e2ee setup`은 계정이 이미 활성 상태면 `E2EE가 이미 활성화되어 있습니다`로 종료한다.
- `toard-shim e2ee approve`는 로컬 credentials의 `content_owner_id`, `content_key_version`과 OS 보안 저장소의 UCK가 이미 있다고 가정한다.
- 기기 승인 요청 생성은 `kind="browser"`만 허용한다.
- 따라서 두 번째 컴퓨터의 shim은 계정의 기존 UCK를 받을 수 없다.
- 두 번째 컴퓨터에서 기존 히스토리를 보는 브라우저 승인과 그 컴퓨터의 프롬프트 수집 활성화가 서로 분리되어 있다.
- 실패 메시지 `수집 자격 증명이 없습니다`는 일반 ingest token 문제처럼 보이지만 실제로는 E2EE 기기 미승인 상태다.

이 구조에서는 사용자가 새 컴퓨터에서 `approve`를 실행해야 하는 것으로 오해하기 쉽고, 계정 E2EE가 활성화된 뒤에도 새 컴퓨터는 `server_v1` 전송을 시도해 `409 E2EE_REQUIRED`를 받는다.

## 3. 목표

- 새 컴퓨터에서 한 번의 연결 흐름으로 shim 수집과 현재 브라우저 열람을 모두 활성화한다.
- 기존 컴퓨터에서 수집한 히스토리를 새 컴퓨터에서 볼 수 있게 한다.
- 새 컴퓨터에서 수집한 히스토리도 기존 승인 기기에서 볼 수 있게 한다.
- 사용자에게 `수집 기기`, `열람 브라우저`, `UCK wrapper` 같은 내부 용어를 요구하지 않는다.
- 서버에 평문 UCK, Recovery Kit, 기기 개인키를 전송하지 않는다.
- 승인 대기 또는 실패 중에는 본문 수집만 보류하고 사용량 수집은 계속한다.
- 중단된 연결을 안전하게 재개하며 새 UCK를 임의로 생성하거나 기존 계정 키를 덮어쓰지 않는다.

## 4. 비목표

- 서로 다른 사용자 간 히스토리 공유
- 팀 공용 콘텐츠 키
- 기기 간 로컬 히스토리 파일 자체 동기화
- UCK 회전과 기존 DEK 전체 재래핑
- 패스키 PRF를 이용한 자동 승인
- 승인된 기기의 원격 파일 또는 운영체제 제어

## 5. 검토한 접근법

### 5.1 컴퓨터별 독립 UCK

각 컴퓨터가 별도 UCK를 생성하고 자기 기록만 복호화한다.

- 장점: 새 컴퓨터 연결이 단순하다.
- 단점: 사용자 단위 히스토리를 만들 수 없고, 기기마다 열람 범위가 갈린다.
- 결정: 채택하지 않는다.

### 5.2 shim만 먼저 승인하고 브라우저는 별도 승인

새 컴퓨터의 shim만 기존 UCK를 받고, 히스토리를 처음 열 때 브라우저 승인을 한 번 더 수행한다.

- 장점: 현재 API를 비교적 적게 변경한다.
- 단점: 사용자가 같은 컴퓨터를 두 번 승인해야 하며 현재의 혼란을 그대로 남긴다.
- 결정: 독립 fallback으로는 유지할 수 있지만 기본 연결 흐름으로 채택하지 않는다.

### 5.3 shim과 현재 브라우저를 하나의 연결 요청으로 승인

새 컴퓨터 연결 페이지가 shim 공개키와 현재 브라우저 공개키를 하나의 연결 요청으로 묶는다. 기존 신뢰 기기는 확인 코드 한 번으로 두 공개키 각각에 UCK를 암호화한다.

- 장점: 사용자 승인 한 번으로 수집과 열람이 모두 준비된다.
- 장점: 서버는 UCK를 계속 알지 못한다.
- 장점: 현재 `content_devices`와 device wrapper 모델을 확장해 사용할 수 있다.
- 단점: 복수 대상 승인 요청과 로컬 shim-브라우저 페어링이 필요하다.
- 결정: 채택한다.

## 6. 제품 개념

### 6.1 사용자에게 보이는 개념

제품의 기본 단위는 **연결된 컴퓨터**다.

- `이 컴퓨터 연결`
- `승인 대기`
- `연결됨`
- `연결 실패`
- `연결 해제됨`

연결 완료 설명은 다음처럼 결과 중심으로 표시한다.

> 이 컴퓨터에서 프롬프트 히스토리를 안전하게 기록하고, 계정의 기존 히스토리를 볼 수 있습니다.

`shim`, `브라우저 키`, `wrapper`, `content_owner_id`는 일반 UI에 표시하지 않는다.

### 6.2 내부 기기 모델

한 물리적 컴퓨터 연결은 내부적으로 두 `content_devices`를 포함할 수 있다.

- `kind="shim"`: 로컬 로그 수집과 암호화 담당
- `kind="browser"`: 현재 브라우저 프로필에서 히스토리 복호화 담당

두 기기는 동일한 `connection_id`로 그룹화한다. 각 기기는 독립 키쌍과 독립 UCK wrapper를 갖는다.

브라우저 프로필이 삭제되거나 다른 브라우저를 사용하면 해당 브라우저만 별도로 다시 승인할 수 있다. 이 경우 제품 문구는 `새 컴퓨터 연결`이 아니라 `이 브라우저에서 히스토리 열기`를 사용한다.

## 7. 통합 연결 흐름

### 7.1 진입

설정의 컴퓨터 연결 화면에서 사용자가 E2EE 히스토리 기록을 선택하면 설치 후 `toard-shim e2ee connect`가 실행된다.

`connect`는 서버 계정 상태에 따라 자동 분기한다.

- E2EE 계정이 `pending` 또는 없음: 기존 최초 설정과 Recovery Kit 발급 수행
- E2EE 계정이 `active`이고 이 shim에 UCK가 없음: 새 컴퓨터 연결 수행
- 로컬 shim에 현재 UCK가 있음: 이미 연결된 상태를 표시하고 종료

기존 `e2ee setup`은 호환용 alias로 유지하되 사용자 안내에서는 `e2ee connect`만 사용한다.

### 7.2 새 컴퓨터의 로컬 준비

1. shim이 새 HPKE 기기 키쌍을 생성한다.
2. shim 개인키는 OS 보안 저장소에 `pending` 상태로 저장한다.
3. shim이 loopback 전용 짧은 수명의 연결 서버를 시작한다.
4. shim이 toard의 `이 컴퓨터 연결` 페이지를 기본 브라우저로 연다.
5. loopback 주소와 일회용 capability는 URL fragment로 전달해 서버 요청과 referrer에 포함하지 않는다.
6. 웹 페이지가 capability를 사용해 로컬 shim 공개키와 기기 라벨을 읽는다.
7. 브라우저는 Web Crypto로 non-extractable 기기 키쌍을 생성한다.

loopback 서버는 `127.0.0.1` 또는 `::1`에만 바인딩하고, 허용된 toard origin에만 CORS 응답하며, capability는 한 번 사용하거나 10분 후 만료한다.

### 7.3 복수 대상 연결 요청

로그인된 브라우저는 서버에 하나의 연결 요청을 생성한다.

- 물리적 컴퓨터 라벨과 플랫폼
- shim 공개키
- 현재 브라우저 공개키
- 요청 생성 브라우저의 로그인 사용자

서버는 같은 사용자 아래에 pending `shim`, `browser` 기기를 생성하고 하나의 `connection_id`, 6자리 확인 코드, 5분 만료 시간을 반환한다.

브라우저와 로컬 shim 화면에는 같은 코드와 기존 신뢰 기기에서 승인하라는 안내를 표시한다.

### 7.4 기존 신뢰 기기 승인

기본 V1 승인자는 현재 UCK를 가진 기존 shim이다.

1. 사용자가 기존 신뢰 컴퓨터에서 `toard-shim e2ee approve`를 실행한다.
2. CLI가 pending 연결 요청의 컴퓨터 라벨, 플랫폼, 생성 시각, 대상 종류를 표시한다.
3. 사용자가 새 컴퓨터에 표시된 6자리 코드를 입력한다.
4. 기존 shim이 로컬 UCK를 읽는다.
5. shim 공개키와 브라우저 공개키 각각에 UCK를 HPKE seal한다.
6. 두 envelope를 하나의 승인 요청으로 서버에 전송한다.
7. 서버는 두 기기를 같은 트랜잭션에서 승인하고 wrapper를 저장한다.

잠금 해제된 브라우저가 새 컴퓨터를 승인하는 기능은 같은 서버 계약을 사용할 수 있지만, V1 필수 범위에는 포함하지 않는다. 먼저 기존 shim 승인 경로를 완성한다.

### 7.5 새 컴퓨터의 완료

승인 후:

- 새 브라우저는 자신의 envelope를 받아 UCK를 복호화하고 IndexedDB에 기기 개인키를 저장한다.
- 새 shim은 자신의 envelope를 받아 UCK를 복호화하고 OS 보안 저장소에 저장한다.
- shim credentials에 `collect_content=e2ee_v1`, `content_owner_id`, `content_key_version`, `content_device_id`, `content_connection_id`를 원자적으로 기록한다.
- 브라우저와 shim이 각각 자기 wrapper를 소비하거나 재조회할 수 있게 한다.
- 두 대상이 준비되면 UI를 `연결됨`으로 전환한다.

연결 완료 직후 새 브라우저는 사용자 계정의 기존 E2EE 히스토리를 조회하고 로컬에서 복호화할 수 있다. 새 shim이 이후 업로드하는 기록도 같은 UCK 버전을 사용하므로 기존 승인 기기에서 볼 수 있다.

## 8. 데이터 모델과 API

### 8.1 연결 요청

복수 대상을 원자적으로 승인하기 위해 연결 요청과 대상을 분리한다.

`content_device_connection_requests`

- `id`
- `user_id`
- `label`, `platform`
- `confirmation_code_hash`
- `created_at`, `expires_at`, `approved_at`

`content_device_connection_targets`

- `request_id`
- `device_id`
- `encapsulated_key`
- `encrypted_envelope`
- `consumed_at`

기존 `content_devices`와 `content_key_wrappers`는 유지한다. `content_devices`에는 nullable `connection_id`를 추가해 같은 컴퓨터의 shim과 브라우저를 묶는다.

기존 단일 브라우저 승인 API는 호환을 위해 유지하고, 신규 통합 연결은 별도 API를 사용한다.

### 8.2 신규 API

브라우저 세션 인증:

- `POST /api/content/connections`
  - shim과 브라우저 공개키를 받아 연결 요청 생성
- `GET /api/content/connections/:id`
  - 현재 연결 요청 상태와 브라우저 envelope 조회

ingest token 인증:

- `POST /api/v1/content/connections`
  - 필요 시 CLI 단독 shim 연결 요청 생성
- `GET /api/v1/content/connections/:id`
  - shim 대상 승인 상태와 envelope 조회
- `POST /api/v1/content/connections/:id/approve`
  - 기존 신뢰 shim이 모든 대상 envelope를 제출
- `GET /api/v1/content/devices/:id/wrapper`
  - 중단 후 재개 시 승인된 shim이 자기 wrapper를 다시 가져옴

모든 API는 사용자 소유권, 대상 공개키, 활성 키 버전, 만료, 미폐기 상태를 검증하고 `Cache-Control: no-store`를 설정한다.

## 9. 중단과 재개

- 연결 요청 승인 전에는 본문 수집을 시작하지 않는다.
- 사용량 이벤트 수집은 계속한다.
- 브라우저를 닫거나 CLI가 종료돼도 pending 기기 개인키는 로컬 보안 저장소에 남겨 같은 요청을 재개할 수 있다.
- 요청이 만료되면 pending 서버 레코드와 로컬 pending 키를 정리하고 새 요청을 만든다.
- 승인 뒤 브라우저만 완료되지 않은 경우 브라우저는 저장된 개인키로 wrapper를 다시 받아 완료한다.
- 승인 뒤 shim만 완료되지 않은 경우 shim은 저장된 pending 개인키로 wrapper를 다시 받아 완료한다.
- 로컬 pending 개인키가 사라졌다면 기존 envelope를 재사용하지 않고 새 기기 연결 요청을 만든다.
- 계정이 이미 `active`인 상태에서 새 컴퓨터가 새 UCK를 생성하거나 `/activate`를 다시 호출하지 않는다.

## 10. 오류 문구

기술 오류 코드를 그대로 노출하지 않는다.

- 로컬 E2EE 메타데이터 없음:
  - `이 컴퓨터는 아직 E2EE 히스토리에 연결되지 않았습니다.`
  - 기본 동작: `이 컴퓨터 연결`
- 승인 대기:
  - `기존에 연결된 컴퓨터에서 승인해 주세요.`
- 승인 만료:
  - `연결 요청이 만료되었습니다. 다시 연결해 주세요.`
- UCK 또는 pending 개인키 없음:
  - `보안 키를 찾을 수 없어 연결을 다시 시작해야 합니다.`
- `409 E2EE_REQUIRED`:
  - `히스토리는 아직 전송하지 않았습니다. 이 컴퓨터 연결을 완료해 주세요.`
  - `사용량은 정상적으로 기록되었습니다.`
- 기존 신뢰 기기 없음:
  - `Recovery Kit로 이 컴퓨터를 연결`

`수집 자격 증명이 없습니다`는 token 자체가 없을 때만 사용하고, E2EE owner metadata 미설정에는 사용하지 않는다.

## 11. 기기 관리

설정의 히스토리 보안에서는 기본적으로 물리적 컴퓨터 단위로 그룹화한다.

- `MacBook Pro · 연결됨`
- `Mac mini · 연결됨`
- 마지막 사용 시각
- `히스토리 기록 및 열람 가능`

세부 정보에서만 shim과 브라우저 프로필을 구분한다.

브라우저만 삭제된 경우 컴퓨터 전체를 끊지 않고 `이 브라우저 다시 연결`을 제공한다. shim을 폐기하면 해당 컴퓨터의 본문 수집은 중단하지만 이미 승인된 브라우저 wrapper는 명시적 전체 연결 해제 전까지 유지할 수 있다.

`이 컴퓨터 연결 해제`는 같은 `connection_id`의 활성 shim·브라우저 wrapper를 함께 폐기한다. 마지막 복구 수단을 제거하려 하면 Recovery Kit 또는 다른 승인 기기가 있는지 확인하고 차단한다.

## 12. 보안 요구사항

- 평문 UCK와 기기 개인키는 서버, 로그, telemetry, 오류 payload에 포함하지 않는다.
- 연결 요청의 공개키는 요청 생성 시 고정하며 승인 시 교체할 수 없다.
- 확인 코드는 `sha256(request_id + ':' + code)` 형태로만 저장한다.
- 승인 API는 요청과 같은 사용자 ingest token만 허용한다.
- 승인자는 현재 활성 키 버전의 UCK를 실제로 보유해야 한다.
- 두 envelope는 같은 요청의 고정된 두 공개키에 대해서만 허용한다.
- 승인과 wrapper 저장은 한 트랜잭션에서 처리한다.
- loopback capability는 URL fragment에만 두고 서버 접근 로그에 남기지 않는다.
- loopback 서버는 로컬 인터페이스에만 바인딩하고 허용 origin, method, body size를 제한한다.
- pending 개인키도 OS 보안 저장소 또는 non-extractable Web Crypto 키로 저장한다.
- 재시도 시 개인키와 공개키가 일치하지 않으면 기존 요청을 폐기한다.

## 13. 테스트 전략

### 13.1 shim

- 계정 `pending`이면 최초 설정, `active`이면 join으로 분기한다.
- 활성 계정의 새 컴퓨터가 새 UCK를 생성하지 않는다.
- 복수 대상 공개키에 각각 올바른 envelope를 만든다.
- pending shim 개인키 저장, 승인 재개, 만료 정리를 검증한다.
- 연결 완료 전 content cursor가 갱신되지 않는다.
- `409 E2EE_REQUIRED`가 연결 안내로 변환되고 사용량 수집은 실패하지 않는다.

### 13.2 서버

- 한 연결 요청에 shim과 browser 두 target이 생성된다.
- 잘못된 코드, 다른 사용자, 만료, 승인 재사용을 거부한다.
- 한 target이 빠진 승인 payload를 거부한다.
- 두 기기 승인과 wrapper 저장이 원자적으로 처리된다.
- target별 envelope는 해당 target만 조회할 수 있다.
- 중단 후 wrapper 재조회가 가능하다.
- 서버 요청과 DB 호출에 평문 UCK가 존재하지 않는다.

### 13.3 브라우저

- loopback capability로 shim 공개키를 가져온 뒤 브라우저 키를 생성한다.
- 승인 완료 후 기존 맥북 히스토리를 복호화한다.
- 새 브라우저가 저장한 device id로 재접속 시 자동 잠금 해제한다.
- 브라우저 개인키 삭제 시 컴퓨터 전체가 아니라 브라우저 재승인을 안내한다.
- 좁은 화면과 한국어·영어에서 연결 상태와 코드를 명확히 표시한다.

### 13.4 E2E

- 맥북 최초 활성화 → 맥미니 연결 요청 → 맥북 승인 → 맥미니 기존 히스토리 열람
- 맥미니 새 프롬프트 수집 → 맥북에서 동일 기록 열람
- 승인 중 브라우저 종료와 CLI 종료 후 재개
- 승인 만료 후 재요청
- 기존 신뢰 기기 없이 Recovery Kit로 새 컴퓨터 연결
- 네트워크 payload와 서버 로그에 평문 본문, UCK, Recovery Kit가 없음

## 14. 출시 순서

1. 복수 대상 연결 요청 데이터 모델과 서버 계약
2. shim `e2ee connect`와 pending 키 저장·재개
3. 연결 페이지의 loopback 페어링과 브라우저 target 생성
4. 기존 shim의 복수 target 승인
5. 새 shim과 브라우저의 envelope 소비 및 완료 상태
6. 컴퓨터 단위 기기 관리와 오류 문구 정리
7. Recovery Kit 기반 새 컴퓨터 연결을 같은 완료 모델에 통합

기존 단일 브라우저 승인 경로와 최초 E2EE 설정은 새 흐름이 검증될 때까지 유지한다.

## 15. 완료 기준

- 사용자는 새 컴퓨터에서 `이 컴퓨터 연결` 한 번과 기존 신뢰 기기 승인 한 번만 수행한다.
- 연결된 맥미니에서 기존 맥북 히스토리를 볼 수 있다.
- 맥미니에서 새로 수집한 기록을 맥북에서도 볼 수 있다.
- 새 컴퓨터 연결 과정에서 새 UCK를 생성하거나 기존 사용자 키를 덮어쓰지 않는다.
- 승인 전에는 본문을 보내지 않으며 사용량 수집은 계속한다.
- 서버는 평문 UCK와 프롬프트 본문을 받지 않는다.
- 중단된 연결을 같은 기기 키로 재개하거나 안전하게 새로 시작할 수 있다.
- 일반 UI에는 수집 기기와 열람 브라우저를 별개 설정으로 요구하지 않는다.
