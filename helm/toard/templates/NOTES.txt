toard 가 배포되었습니다. (release: {{ .Release.Name }}, ns: {{ .Release.Namespace }})

1) 파드 상태:
   kubectl get pods -n {{ .Release.Namespace }} -l app.kubernetes.io/instance={{ .Release.Name }}

2) 접속:
{{- if .Values.ingress.enabled }}
   https://{{ .Values.ingress.host }}  (Ingress)
{{- else }}
   포트포워드: kubectl -n {{ .Release.Namespace }} port-forward svc/{{ include "toard.fullname" . }} 3000:{{ .Values.service.port }}
   → http://localhost:3000
{{- end }}

3) 로그인:
{{- if eq .Values.config.authMode "open" }}
   AUTH_MODE=open — 인증 없이 접근(내부망 전제).
{{- else }}
   /login (id/pw) · /signup (가입){{ if .Values.config.allowedEmailDomains }} · 허용 도메인: {{ .Values.config.allowedEmailDomains }}{{ end }}
{{- if not .Values.migrate.seedOnInstall }}
   최초 admin: migrate.seedOnInstall=true + secrets.bootstrapAdmin.* 로 재설치하거나,
   seed Job 을 수동 실행하세요.
{{- end }}
{{- end }}

4) schema migration:
{{- if .Values.migrate.enabled }}
   일반 Job {{ include "toard.migrationJobName" . }} 이 Release.Revision마다 새 이름으로 실행됩니다.
   완료 대기: kubectl wait -n {{ .Release.Namespace }} --for=condition=complete --timeout=10m job/{{ include "toard.migrationJobName" . }}
   install/upgrade 명령 자체도 기다리게 하려면 --wait --wait-for-jobs 를 함께 사용하세요.
   이 옵션이 없어도 app의 /api/ready는 schema 부재·migration 실패 시 503을 반환해 traffic을 차단합니다.
   RollingUpdate maxUnavailable=0 이므로 upgrade 중 기존 ready replica는 새 replica가 준비될 때까지 유지됩니다.
{{- else }}
   migrate.enabled=false — operator가 호환 schema를 먼저 적용해야 합니다.
{{- end }}
